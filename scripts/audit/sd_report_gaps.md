# System Design — missing topics

Surveyed `sd-classic` (16 lessons) and `sd-fde` (8 lessons). The two modules cover the classic SDE-interview canon (URL shortener, rate limiter, feed fan-out, caching, chat, consistent hashing, CAP/PACELC, BoE, B-tree/LSM, Raft, reliability patterns, Kafka, geo-dispatch, payments, notifications) plus the FDE-flavored layer (multi-tenancy, webhooks, SSO, VPC deploys, event sourcing/CQRS, gRPC vs REST vs GraphQL, legacy-ERP integration ladder, customer-facing observability).

What's NOT covered but a senior FDE/SDE candidate is expected to handle in a 2026 interview:

| Topic | Suggested module | Suggested lesson name | Why it matters in 2026 |
|---|---|---|---|
| CRDTs vs Operational Transformation | sd-classic | `sd-crdt` Realtime collab — CRDTs vs OT (Figma, Linear, Notion) | Multiplayer/realtime is now a default product expectation; Figma/Linear/Notion-style prompts are standard at senior rounds, and "explain CRDT vs OT and when each wins" is the discriminator question. |
| Cell-based architecture | sd-classic | `sd-cell` Cell-based architecture — blast-radius reduction | AWS, Slack, DoorDash, Roblox have all publicly migrated to cells (2024–25). Interviewers now expect "how do you bound blast radius?" as a follow-up to any large-scale design. |
| Tail latency engineering | sd-classic | `sd-tail` Tail latency — hedged requests, p99 vs avg, the "Tail at Scale" toolkit | p99 (not avg) is how senior engineers reason about latency. Hedged requests, request cancellation, and tied requests come up whenever the interviewer pushes "what if one replica is slow?" |
| Backpressure + load shedding | sd-classic | `sd-bp` Backpressure & load shedding — when to drop, when to queue | Reliability rounds in 2026 expect you to name shed strategies (priority-based, adaptive concurrency, Little's Law) — not just "we have a queue." Modern systems explicitly model load shedding alongside circuit breakers. |
| Outbox pattern + CDC | sd-classic | `sd-outbox` Transactional outbox + CDC (Debezium-style) | The dual-write problem comes up in every "design X that publishes events" prompt. Outbox + Debezium / WAL-tail is now the standard answer; interviewers reject "we'll publish to Kafka after the DB commit." |
| Modern queue/streaming landscape | sd-classic | `sd-queues` Queue systems compared — SQS, Kafka, Kinesis, Redis Streams, NATS, Warpstream | "Which queue?" is a routine senior-level question. Candidates who only know Kafka miss SQS (simple work-queue), Kinesis (managed Kafka-ish), Redis Streams (cheap), NATS JetStream, and the cost-disruptive Warpstream/AutoMQ (Kafka-on-S3). |
| Multi-region active-active | sd-classic | `sd-multiregion` Multi-region active-active — Spanner, CockroachDB, write-anywhere tradeoffs | "Design for two regions" is the default escalation in senior rounds. Need to name: leader-per-row vs global consensus, HLC vs TrueTime, latency cost of cross-region quorum, and conflict resolution. |
| Distributed tracing fundamentals | sd-classic | `sd-tracing` Distributed tracing — OpenTelemetry, W3C trace context, sampling strategies | OpenTelemetry hit stable in 2024–25 and is now the standard. Senior engineers must name traceparent, head vs tail sampling, span attributes, and the "we sample 1% but keep 100% of errors" pattern. |
| Service mesh decision | sd-classic | `sd-mesh` Service mesh — Istio ambient vs Linkerd vs "do you need one?" | Istio 1.24 stabilized Ambient (sidecarless) mode; "should we adopt a service mesh?" is now a real architectural question, not a given. Interviewers want the explicit YES/NO tradeoff with mTLS, observability, retries. |
| Edge computing patterns | sd-classic | `sd-edge` Edge compute — Cloudflare Workers, edge KV vs Durable Objects vs region-pinned | Workers + KV + D1 + Durable Objects is a coherent platform story now standard at FDE-style interviews. "Where should this code run — edge, region, or per-user shard?" is a senior judgment call. |
| WebSocket / SSE / WebTransport | sd-classic | `sd-realtime` Realtime transports — WebSocket, SSE, WebTransport, long-poll | Chat-style lesson covers WS at high level, but interviewers now drill the actual transport choice: SSE for one-way, WS for bidi, WebTransport (Chrome-only in 2026) for low-latency UDP-ish, and when each fails. |
| Zero-downtime DB migrations | sd-classic | `sd-migrate` Zero-downtime schema migrations — expand-contract, dual-write, backfill | Every senior FDE prompt eventually hits "you need to change the schema of a billing table that gets 10k writes/sec, with no downtime." Expand-contract + dual-write + shadow-read + cutover is the named pattern (Stripe-style, gh-ost, pg_repack). |
| Cost engineering as a constraint | sd-classic | `sd-cost` FinOps in system design — egress, S3 classes, GPU placement | "Design X" interviews in 2026 increasingly include "estimate the monthly bill." Egress dominates analytics/AI bills; S3 tiering and CloudFront-vs-direct decisions are first-class. Shift-left costing is a Staff+ signal. |
| Fine-grained authorization | sd-fde | `fd-authz` Fine-grained authorization — RBAC vs ReBAC, Zanzibar, OpenFGA, Cedar | "Design Google Drive's sharing model" is a classic prompt and the modern answer is Zanzibar-style ReBAC (OpenFGA) or policy-based (Cedar/OPA). Enterprise customers now require this granularity; SSO+roles is no longer enough. |
| Feature flags + progressive delivery | sd-fde | `fd-flags` Feature flags & progressive delivery — canary, ring, LD/OpenFeature | LaunchDarkly, Statsig, and the OpenFeature CNCF standard are now the default release-control story. Senior FDEs are expected to design rollouts (canary → ring → GA) with kill switches as a deployment safety primitive. |
| AI-augmented system design | sd-fde | `fd-aiaug` AI-augmented systems — vector index alongside primary DB, embedding latency budgets, agent-call observability | AI features are now expected in every system; interviewers want "how do you bolt a vector store next to your primary DB without making writes slow?" and "what's your embedding refresh / staleness policy?" Cross-cuts with AI category but the *system-design shape* (latency budget, fallback to non-AI path, cost cap per request) belongs here. |

Recommended 16 new lessons across modules.
